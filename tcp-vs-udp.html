What Is TCP? TCP vs UDP Explained

TCP and UDP are the two transport layer protocols that carry almost all internet traffic. Understanding exactly how each works ‚Äî and why the A+ exam cares ‚Äî starts here.

Almost every piece of data traveling across a network uses one of two transport layer protocols: TCP (Transmission Control Protocol) or UDP (User Datagram Protocol). They both move data from one device to another, but they do it in fundamentally different ways ‚Äî and knowing which protocol a service uses, and why, is core CompTIA A+ exam knowledge.

TCP

Transmission Control Protocol

  • Establishes a connection before sending data
  • Guarantees every packet arrives
  • Re-sends any lost or corrupted packets
  • Delivers data in the correct order
  • Slower ‚Äî more overhead, more reliable
üì¨ Like a certified letter ‚Äî you get a confirmation it arrived, and if it didn't, it gets sent again.
UDP

User Datagram Protocol

  • No connection established ‚Äî fire and forget
  • No delivery guarantee
  • No retransmission of lost packets
  • No ordering of packets
  • Faster ‚Äî less overhead, less reliable
üì¢ Like a radio broadcast ‚Äî fast and reaches everyone, but no way to know if anyone actually received it.

The TCP Three-Way Handshake

Before TCP sends any data, it establishes a connection using a three-way handshake. This is one of the most tested concepts on the CompTIA A+ exam ‚Äî know the three steps and what each one means.

1
SYN Client ‚Üí Server
The client sends a SYN (synchronize) packet to the server, saying: "I want to establish a connection."

2
SYN-ACK Server ‚Üí Client
The server responds with a SYN-ACK (synchronize-acknowledge), saying: "I received your request and I'm ready."

The client sends an ACK (acknowledge), saying: "Confirmed. Connection established ‚Äî let's transfer data."

The sequence is always SYN ‚Üí SYN-ACK ‚Üí ACK. The exam may ask you to identify what happens at each step, or which step fails when a connection can't be established. If a server is unreachable, the SYN is sent but the SYN-ACK never arrives ‚Äî the connection times out at step 1.

When to use TCP vs UDP

The choice between TCP and UDP comes down to one question: is it worse to be slow or to lose data?

Use TCP when the data must arrive complete and correct ‚Äî loading a webpage, downloading a file, sending an email, accessing a remote desktop. A missing chunk of a file or a half-loaded webpage is useless, so TCP's retransmission guarantees are worth the extra overhead.

Use UDP when speed matters more than perfection ‚Äî live video calls, online gaming, DNS lookups, streaming. In a video call, a dropped frame is barely noticeable. But a 2-second pause while TCP retransmits a lost packet would make the call unusable. UDP just moves on and keeps the stream flowing.

Key Takeaways

TCP = reliable, ordered, connection-based. Uses a three-way handshake (SYN ‚Üí SYN-ACK ‚Üí ACK) before any data transfers
UDP = fast, connectionless, no guarantees. Sends data without establishing a connection ‚Äî packets may arrive out of order or not at all
Both operate at Layer 4 (Transport) of the OSI model
TCP ports: HTTP (80), HTTPS (443), FTP (20/21), SSH (22), SMTP (25), RDP (3389)
UDP ports: DNS (53), DHCP (67/68), SNMP (161/162), TFTP (69)
DNS uses UDP for queries but TCP for zone transfers ‚Äî the exam tests this distinction
